理论部分

Flask框架

一个用Python编写的轻量级Web应用框架，扩展性强，便于快速搭建Web应用的基本功能

Flask(__name__)

  • 为什么使用 __name__？

    这是一种常见用法，允许模块在被导入时和作为脚本直接运行时有不同的行为。当模块被导入时，__name__ 通常是模块的名字；而当模块作为脚本直接运行时，Python 会将 __name__ 设置为 '__main__'。

    这种方式使得模块可以既作为库被其他模块导入使用，也可以作为一个独立的程序运行。

request.args

是 Flask 提供的一个全局变量，它是一个不可变的多字典，包含了所有通过查询字符串传递的参数

request.args.to_dict() 方法将其转换为一个普通的字典，其中的键是参数名称，值是参数值的列表。

配置

通过app.config设置数据库连接字符串和其他配置

初始化

创建SQLAlchemy对象，并将其初始化到Flask应用中

刷新

在 with app.app_context(): 语句块使用 db.drop_all() 和 db.create_all() 等函数实现数据库状态刷新

SQLAlchemy ORM库

一个***SQL（结构化查询语言）工具包和对象关系映射（ORM）框架***，提供了一个高层的ORM以及底层的SQL表达式语言。可以用Python代码来操作数据库，而不必编写原始的SQL语句

ORM实现

创建模型以实现ORM的方法很多，通常用以下两种：

  • db.Model

  • Base

  • 各自适用场景

    • 大型项目或需要高度结构化代码的项目可能会更倾向于使用Base
    • 快速开发或小型项目可能会更倾向于使用db.Model以提高开发速度

  • (非官方但常用)***to_dict方法***：通常定义在模型类内部，用于将模型的实例转换成一个Python字典，实现数据的序列化，方便数据传递

  • 添加索引

    Index() 第一个参数为索引命名，后一个参数指定索引字段

映射关系

  • 一对多、多对一

  • 一对一

    在relationship里再添加参数 uselist=False

  • 多对多

    需要一个中间关联表,通过参数 secondary 来指定

数据库引擎

  • 创建：使用**create_engine函数**来创建一个连接到数据库的引擎

  • 连接与SQL操作

    • 使用 **with engine.connect() as connect: 语句块 + connect.execute() 函数**执行实际的数据库DDL（数据定义语言）操作

    • 使用 engine.connect() + connection.execute

会话（Session）

与数据库交互的对象，可以借其更便捷地执行数据库事务

  • 通过sessionmaker创建会话

  • 常用CURD操作

    • session.commit()

      • 创建或更新一个模型实例时，这些更改最初是在会话的暂存区中进行的，并没有立即写入数据库，调用commit()方法，会话才会执行所有挂起的INSERT、UPDATE或DELETE语句，将这些更改实际应用到数据库中

      • 如果在commit()调用过程中发生任何错误，事务可以回滚，以保持数据库状态的一致性

engine.connect() + connect.execute() VS session

  • ORM操作：如果需要使用ORM特性，如自动将Python对象转换为数据库记录，进行日常的CURD操作，使用session是更合适的选择
  • 原生SQL：如果需要执行ORM不支持的SQL操作，或者想要更直接地控制SQL执行或修改底层数据库架构，那么使用engine.connect()和connect.execute()会更合适
  • 抽象层次：session提供了一个更高层次的抽象，隐藏了SQL的复杂性，而engine.connect()则提供了直接执行SQL的能力

和SQL相关的常用函数

  • **text 函数**：用于创建一个原生的SQL文本表达式，用于执行那些SQLAlchemy不直接支持的SQL语句。

    • 常用于定义触发器的逻辑，因为触发器不是通过ORM映射的，而是直接在数据库层面上定义的

    • 通常与connect.execute()（执行一个SQL语句）[结合使用]

RESTful API 实现

强调组件间的 统一接口 是 REST 架构风格与其他基于网络的风格区分开来的核心特征

路径名称避免动词

使用 HTTP 方法来表达资源操作行为，而不是将行为动词定义到路径中。

  • GET 获取指定 URI 的资源信息
  • POST 通过指定的 URI 创建资源
  • PUT 创建或全量替换指定 URI 上的资源
  • PATCH 执行一个资源的部分更新
  • DELETE 通过指定的 URI 移除资源

HTTP 状态码

大致可以分为以下几类：

状态码 含义
2xx 成功，操作被成功接收并处理
3xx 重定向，需要进一步的操作以完成请求
4xx 客户端错误，请求包含语法错误或无法完成请求
5xx 服务器错误，服务器在处理请求的过程中发生了错误

使用标准状态码，开发人员可以立即识别问题，减少发现不同类型错误的时间

API格式响应

请求以及响应的数据格式用的最多的是JSON格式

路由

当 Flask 应用接收到一个请求时，它会检查 URL 路径和请求方法，找到匹配的路由，并调用相应的视图函数

  • 路由装饰器@app.route
    • 将视图函数映射到一个 URL 路径，告诉 Flask 应用装饰的函数应该处理哪个 URL 路径的请求
  • 视图函数
    • 紧跟在 @app.route 后的用def定义的函数
    • 返回响应时常用 flask.jsonify() 将 Python 对象转换为 JSON 格式的响应体
    • 应该处理所有预期的错误情况，并返回适当的 HTTP 状态码

游标

SQLAlchemy 中通常不推荐直接使用 cursor 和原生 SQL 语句来执行数据库操作，因为这绕过了 ORM 的安全和便利性，但这里还是学习一下游标的基本原理

可以通过游标对象执行SQL语句，在一些场景下提升性能

  • 执行查询后，游标允许开发者逐行读取结果集，或者直接获取全部结果
  • 适用场景：逐条处理数据、批量处理数据、事务处理（与事务控制语句如BEGIN、COMMIT、ROLLBACK一起使用）

python动态访问和赋值函数（SQLAlchemy实现）

  • getattr(object, name, default) 获取对象的属性值

    1. object：要获取属性的对象。
    2. name：属性的名称，通常是一个字符串。
    3. default：（可选）如果属性 name 不存在于 object 中，则返回此默认值。

  • setattr(object, name, value) 设置对象的属性值

    1. object：要设置属性的对象。
    2. name：要设置的属性的名称，通常是一个字符串。
    3. value：要设置的属性值。

实验难点（开发过程中遇到的困难）

Test1：ORM实现

映射关系的建立

因为在之前常接触的数据库操作中，一对多关系通常只用在多端定义外键即可，而这次给出的department模型类中就定义了一对多关系。这让我一度很迷惑。一个关系要定义几次？有哪些地方需要注意？

查找资料发现，这部分的实现方式有很多种，不同人的习惯不同，写法也不同，没有一个所谓的标准答案。费了些劲，才总结出一套自己使用比较顺畅的方法。

  • 一对多、多对一

  • 一对一

    在relationship里再添加参数 uselist=False

  • 多对多

    需要一个中间关联表,通过参数 secondary 来指定

engine.connect() + connect.execute() VS session

最开始，我认为两者的作用是一样的，但提供的代码中这两种操作都有，于是很疑惑为什么不统一用一种操作？

看了一些代码后发现，两者确实功能近似，但有着各自的优势适用场景

  • ORM操作：如果需要使用ORM特性，如自动将Python对象转换为数据库记录，进行日常的CURD操作，使用session是更合适的选择
  • 原生SQL：如果需要执行ORM不支持的SQL操作，或者想要更直接地控制SQL执行或修改底层数据库架构，那么使用engine.connect()和connect.execute()会更合适
  • 抽象层次：session提供了一个更高层次的抽象，隐藏了SQL的复杂性，而engine.connect()则提供了直接执行SQL的能力

Test2：RESTfulAPI实现

model使用

（这部分应该是完成这次实验中最痛苦的一部分 😢）

使用 SQLALCHEMY 实现RESTfulAPI，对已建立的模型类的使用十分关键，需要利用其来创建对象实例、获取主键等等。

这次实验中为了正确使用已建立的模型类踩了不少坑

  • 问题1——使用Model = getattr(db.models, table_name.capitalize(), None)报错

    这是在网络上看到的一个用法，一度以为是正确的。最开始没有怀疑这个用法有误，耽误了很多时间。

    但后面发现 “SQLAlchemy 会自动将 User 类注册到 db.models 模块中”这一说法根本没有官方文档支撑，目前个人认为比较合理的解释是手动定义该字段，实际模型还是需要自己导入或者创建。

    • 解决方法

      从Test1-ORM.py导入相关模块

  • 问题2——在使用 RESTful API 修改数据后，发现数据库中的修改在重新运行脚本时消失了

    • 失败思路1

      一开始以为是接口实现出了问题，比如是不是没有提交数据。但后面一一排除了这些可能，确定bug不在于接口实现本身

    • 失败思路2

      后来考虑到是不是因为从 Test1_ORM import app, db ，而Test1_ORM 中有 db.drop_all()，使用它的app、db实例会触发整体数据库的重建。

      于是尝试了在 Test2-RESTful.py 中重新创建app、db实例，但仍未解决问题

    ……

    经过很久的探索，终于发现，症结确实如基本失败思路2中所想——Test1_ORM 中有 db.drop_all()，会触发整体数据库的重建。但认识不够准确，并不只是使用它的app、db实例会触发整体数据库的重建，因为模型类的创建也需要使用db.Model作为示例，同样会触发db.drop_all()。所以不能从Test1_ORM import，于是最终把模型类也在Test2-RESTful.py 中重新定义了一次。

    • 为什么不删除Test1_ORM 中 db.drop_all()？

      怕因为改变了这里，检查作业时多次执行脚本的时候出问题。
