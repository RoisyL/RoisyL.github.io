概念与思想

DevOps

是一种软件开发方法（或者说软件工程思想），将持续开发、持续测试、持续集成、持续部署和持续监控贯穿于软件开发的整个生命周期

CI/CD 流水线

通过自动化流程和工具自动构建应用、测试应用、部署应用

CI

CI 代表持续集成（Continuous Integration），是指开发团队通过版本控制工具，将代码的不同部分集成到共享存储库中，并利用自动化构建工具和单元测试框架频繁地进行构建和测试

CD

CD 代表持续交付（Continuous Delivery）和持续部署（Continuous Deployment），建立在持续集成的基础上，运行脚本构建打包应用，通过自动化部署工具部署到生产环境运行应用，监控生产环境指标，确保软件可以随时随地进行部署与应用

自动化工具

容器化技术栈

Docker

一个开源的容器化平台，用于创建、管理虚拟环境（容器）

Kubernetes（K8s）

一个开源的容器编排系统，用于自动化部署、扩展和管理容器化应用程序

  • Docker 提供了容器化技术的基础，而***Kubernetes*** 在此基础上提供了容器的高级编排和管理功能。在实际应用中，它们经常一起使用，以实现高效的容器化应用部署和管理。

CI/CD技术栈

Jenkins

提供各种接口和工具，以实现持续集成和持续交付流程的自动化

GitHub Actions

一个实现持续集成交付的自动化流程工具

  • 核心：通过编写 YAML 文件， 可以定义在代码库中发生特定事件(比如：Push、Tag等) 时所触发的工作流程，从而实现自动化的 CI/CD

GitHub Actions

基本概念

  • 工作流（Workflows）

    工作流是GitHub Actions执行任务的基本单位，可以为Git上不同的事件（如push、pull、request等）定义不同的工作流，以响应操作代码的变更

  • 任务（Jobs）

    工作流程由一个或多个任务组成，每个任务运行在独立的虚拟环境中。任务可以是构建、测试、部署等操作

  • 步骤（Steps）

    任务由多个步骤组成，每个步骤执行一个操作。一个步骤可以是运行命令、使用某个预定义的操作，或者调用自定义脚本

  • 动作（actions）

    每个 step 可以依次执行一个或多个命令

workflow文件基础结构

  • name

    指定 workflow 的名称

  • on

    指定触发 workflow 的条件，通常是某些事件。

    • on字段也可以是事件的数组。

      完整事件列表见官方文档。除了代码库事件，GitHub Actions 也支持外部事件触发，或者定时运行。

    • on.<push|pull_request>.<tags|branches>

      指定触发事件时，可以限定分支或标签。

  • jobs.<job_id>.name

    workflow 文件的主体是jobs字段，表示要执行的一项或多项任务。

    jobs字段里面，需要写出每一项任务的job_id，具体名称自定义。job_id里面的name字段是任务的说明。

  • jobs.<job_id>.needs

    指定当前任务的依赖关系，即运行顺序。

  • jobs.<job_id>.runs-on

    指定运行所需要的虚拟机环境。它是必填字段

  • jobs.<job_id>.steps

    指定每个 Job 的运行步骤，可以包含一个或多个步骤。每个步骤都可以指定以下三个字段。

部署实操

踩坑

  • 在部署新版本的应用程序或执行其他维护任务之前，确保相关旧服务已经被正确地停止

  • 运行需要持续运作的脚本时最好使用 nohup ，它是一个命令行实用程序，用于运行命令，使得该命令在用户注销或终端关闭后继续运行

  • 构建新镜像之前可以先删掉旧镜像，以免覆盖出问题

常用命令行工具

sshpass

一个非交互式的 ssh 密码提供工具，用于在命令行中直接提供 SSH 密码

  • -p "$SSH_PASSWORD"：指定要使用的 SSH 密码

rsync

一种快速、多功能的文件复制工具。它可以用来本地复制、到/从远程服务器复制文件或在远程服务器之间复制文件。

  • -a 代表 “archive”，它保留了文件的权限、属性，并递归地复制目录内容
  • -v 代表 “verbose”，它会在执行过程中显示详细的信息
  • -z 代表 “compress”，它在传输文件时对数据进行压缩，以加快传输速度并减少带宽消耗
  • -e "ssh -o StrictHostKeyChecking=no"：指定 rsync 使用 SSH 作为传输层，并通过 -o StrictHostKeyChecking=no 告诉 SSH 不要进行严格的主机密钥检查。这意味着它会接受任何主机密钥，这在自动化脚本中很常见，但可能带来安全风险，因为它容易受到中间人攻击。
