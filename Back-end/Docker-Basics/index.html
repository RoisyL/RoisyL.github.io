docker run（以创建MySQL容器为例）

安装命令解读

docker run -d：创建并运行一个容器，-d参数表示后台运行

--name mysql：容器别名

-p 3306:3306：端口映射，宿主机端口:容器内端口

  • 容器内端口由容器内的进程决定，MySQL默认端口是3306；宿主机端口可任意指定，不冲突即可，通常与容器内端口一致

-e key:value：配置容器内的一些参数，这是通用格式

  • TZ：指定时区
  • MYSQL_ROOT_PASSWORD：指定mysql密码

mysql：镜像的名字，Docker会根据这个名字搜索并下载，不可随意设置

  • 完整格式：镜像名字:版本
  • 该写法默认等同于：mysql:latest，即mysql镜像的最新版本

Docker数据卷

容器提供程序的运行环境，但是程序运行产生的数据、程序运行依赖的配置都应该与容器解耦，通过数据卷可以实现该功能

概念

*数据卷(volume)*是一个虚拟目录，是容器内目录与宿主机目录之间映射的桥梁。

数据卷关联容器内的特定目录，同时指向宿主机的特定目录，这样宿主机和容器的目录就关联起来了，称为***挂载***

常用命令

命令 说明 文档地址
docker volume create 创建数据卷 docker volume create
docker volume ls 查看所有数据卷 docs.docker.com
docker volume rm 删除指定数据卷 docs.docker.com
docker volume inspect 查看某个数据卷的详情 docs.docker.com
docker volume prune 清除数据卷 docker volume prune

  • 默认数据卷所在目录：/var/lib/docker/volumes/数据卷名字/_data

  • 容器与数据卷的挂载要在创建容器时配置，这一过程中，数据卷会默认自动创建

    • 匿名卷

      当创建容器的时候，未指定数据卷，就会创建一个匿名卷，卷名由一串hash值组成

  • eg. nginx指定数据卷安装

挂载本地目录或文件

数据卷的目录结构较深，直接操作会不太方便。在很多情况下，我们会直接将容器目录与宿主机指定目录挂载

注意：本地目录或文件，必须以/或./开头，即以绝对路径或相对路径开头。若以名字开头，会被认为是数据卷名而非本地目录名

自定义镜像（以创建java应用镜像为例）

镜像结构

镜像搭建步骤

自定义镜像本质就是依次准备好程序运行的基础环境、依赖、应用本身、运行配置等文件，并且打包而成。

部署一个java应用，大致流程如下：

  • 准备Linux服务器
  • 安装并配置JDK
  • 上传jar包
  • 运行jar包

对应自定义镜像大概也是如下：

  • 准备Linux运行环境（非全部，仅基础运行环境即可）
  • 安装并配置JDK
  • 拷贝jar包
  • 配置启动脚本

“层”

上述步骤中的每一次操作其实都是在生产一些文件（系统运行环境、函数库、配置最终都是磁盘文件），所以镜像就是一堆文件的集合

镜像文件按照操作的步骤分层叠加而成，每一层形成的文件都会单独打包并标记一个唯一id，称为Layer（层）

如果构建时用到的某些层已经制作过，就可以直接拷贝使用这些层

镜像结构文件：Dockerfile

官方文档：https://docs.docker.com/engine/reference/builder/

由于制作镜像需要逐层处理和打包，Docker提供自动打包的功能。只需要将打包的过程，每一层要做的事情用固定的语法写下来，交给Docker去执行即可。

这种记录镜像结构的文件就称为***Dockerfile***

常用命令

指令 说明 示例
FROM 指定基础镜像 FROM centos:6
ENV 设置环境变量，可在后面指令使用 ENV key value
COPY 拷贝本地文件到镜像的指定目录，并改名 COPY ./xx.jar /tmp/app.jar
ADD 复制新文件或目录到容器内，也可以从URL下载文件到容器内（与COPY 相比，适用于需要自动解压缩或下载远程文件的场景） ADD <源路径> <目标路径>
RUN 执行Linux的shell命令，一般是安装过程的命令 RUN yum install gcc
EXPOSE 声明容器运行时监听的端口，是提示镜像使用者的 EXPOSE 8080
CMD 定义容器的默认执行命令 CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
ENTRYPOINT 镜像中应用的启动命令，容器运行时调用 ENTRYPOINT java -jar xx.jar
WORKDIR 指定工作目录，后续的指令都会在这个目录下执行 WORKDIR /code

  • CMD 可以与 ENTRYPOINT 指令结合使用。ENTRYPOINT 指定了容器启动时要运行的可执行文件，而 CMD 提供了传递给这个可执行文件的参数

Docker网络

什么是Docker网络？

一个java项目往往需要访问各种中间件，例如Redis、MySQL等，容器之间是可以相互访问的

不使用网络的话，只能用容器实例的ip+端口来访问，但是这里的ip是虚拟的，不固定。如果在开发时候写死ip，往往会导致部署的时候连接失败。

Docker网络就可以用来解决这个问题，建立Docker网络后将容器加载进来，就可以通过容器名字访问

只有在相同的网络之中才能进行通讯

docker network disconnect 使指定容器连接离开某网络 docker network disconnect

Docker-compose

在线上环境中，通常不会将项目的所有组件放到同一个容器中；更好的做法是把每个独立的功能装进单独的容器
同一个服务器上有可能会运行着多个容器，如果每次都靠一条条指令去启动太繁琐。 Docker-compose 就是解决这个问题的，它用来编排多个容器，将启动容器的命令统一写到 docker-compose.yml 文件中，以后每次启动这一组容器时，只需要 docker-compose up 就可以了

docker-compose.yml 示例

version 代表 docker-compose.yml 的版本，目前最新版为 3，不需要改动它。

services定义了所有参与的容器服务，每个服务下可以有自己的配置。定义了一个名叫 app 的容器。后面的内容都是 app 容器的相关配置：

  • restart ：除正常工作外，容器会在任何时候重启，比如遭遇 bug、进程崩溃、docker 重启等情况。

  • build ：指定一个包含 Dockerfile 的路径，并通过此 Dockerfile 来构建容器镜像

    • 也可以用 image: xxx 指定镜像，但这样就需要先手动构建好镜像

  • command ：容器运行时需要执行的命令

  • volumes ：卷，或者说是挂载

  • ports ：定义宿主机和容器的端口映射

  • environment：定义环境变量

  • depends_on：声明该服务依赖于其他服务，确保其他服务先启动

  • container_name：为服务容器指定自定义名称
