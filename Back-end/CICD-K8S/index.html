简介

K8s

Kubernetes（或 K8s）是用于管理容器化工作负载和服务的便携式开源平台。它有助于在集群中编排和管理 pod 中的应用程序

K3s

K3s 是CNCF认证的 Kubernetes 轻量级发行版，仍可以编写 YAML 来像使用 K8s 一样在 K3s 集群上运行，专为低资源环境而设计。

K8s的使用和配置比较复杂，对于小型开发/测试集群而言，使用K3s可以在一定程度上简化配置与使用。

为什么需要K8s

在实际生产环境中，

  1. 通常一个应用包含多个容器：

    1. 最简单的Web应用：数据库容器+后端应用容器+前端应用容器（通常是一个Nginx容器）
    2. 微服务场景中，后端可能涉及多个微服务模块，每个模块对应一个容器

  2. 每个类别的服务通常会有多个容器实例。

    对于一些负载较高的服务，通常会部署多个相同的容器实例达到负载均衡的效果，从而提高服务整体的吞吐量。

  3. 容器是不稳定的，需要时刻监控容器的状态

仅靠docker命令难以完成上述任务的，需要一个专门的容器编排调度工具处理

重要概念

基础元素

集群与节点

在 k3s 中，“集群”（Cluster）指的是一组工作在一起的服务器（节点），它们共同提供容器化应用的运行环境。

  • 集群中的角色

    通常分为两类：控制平面节点（运行 k3s 控制服务的节点）和工作节点（运行应用容器的节点）。

    • 控制平面 control-plane/master

      k3s 集群的大脑，负责 API 服务、调度、集群状态管理等

    • 工作节点 worker

      运行容器化应用的地方。它们通过与控制平面通信来接收指令和管理容器

  • 配置文件路径

    配置文件（通常命名为 kubeconfig）包含了访问集群所需的认证信息、配置上下文、集群地址等关键信息。配置文件的路径决定了 kubectl 命令行工具如何识别和访问 Kubernetes 集群，因此初始建立集群时，我们需要定义一个配置文件位置。

容器化

  • Pod

    是 Kubernetes 中的基本部署单元，它是一个或多个容器的集合，这些容器被设计为一起工作，它们共享网络和存储资源。Pods 使得容器可以协同运行，例如，一个 Web 应用的前端和后端可以部署在同一个 Pod 中。

  • Kubernetes创建容器和使用Docker创建容器的异同

Pod Controller

背景

  • container是脆弱的。在实际的生产环境中，container中运行的进程很可能因为各种各样的原因挂掉。这时候，快速恢复业务的方法是重新启动一个新的容器实例
  • 另一方面，为了实现负载均衡或并行计算，我们需要维护相同的多个容器实例，来共同完成任务

归结起来可以表示为：我们需要在集群中维护一定数量的容器实例。Pod Controller 就是达到这一目的的方法。

Controller思想

Kubernetes中的控制器（Controller）将会监控当前集群中的状态，并努力使集群的当前状态满足用户设置的目标状态

*其他常见的Pod Controller

  • Job

    创建一个或多个Pod，并确保指定数量的Pod成功终止。

    Job一般用于定义并启动一个批处理任务。批处理任务通常并行（或串行）启动多个计算进程去处理一批工作项，处理完成后，整个批处理任务结束。

    • Kubernetes支持以下几种Job:
      • 非并行Job: 通常创建一个Pod直至其成功结束
      • 固定结束次数的Job: 设置.spec.completions，创建多个Pod，直到.spec.completions个Pod成功结束
      • 带有工作队列的并行Job: 设置.spec.Parallelism但不设置.spec.completions，当所有Pod结束并且至少一个成功时，Job就认为是成功。

  • DaemonSet

    用于管理在集群中每个Node上运行且仅运行一份Pod的副本实例。

    • 常见使用场景：在每个Node上都运行一个存储进程/一个日志采集程序/一个性能监控程序

  • StatefulSet

    用来搭建有状态的应用集群（比如MySQL、MongoDB等），保证StatefulSet中各应用实例在创建和运行的过程中，都具有固定的身份标识和独立的后端存储；还支持在运行时对集群规模进行扩容、保障集群的高可用等功能。

Service

Service可以将运行在一组Pods上的应用程序公开为网络服务，简单地实现服务发现、负载均衡等功能。

背景

k8s的Pods具有自己的生命周期，同一时刻运行的Pod集合与稍后运行的Pod集合很有可能不同（如发生更新、node故障等），Pods的IP地址可能会随时发生变化。

这就会导致一个问题：如果一组后端Pods为集群内其他前端Pods提供功能，那么前端Pods该如何找出并跟踪需要连接的IP地址？通过Service，能够解耦这种关联，方便的通过Service地址访问到相应的Pods，前端不应该也没必要知道怎么访问、访问到的具体是哪一个Pod。

Service的4种类型

  • ClusterIP：通过集群的内部 IP 暴露服务，选择该值，服务只能够在集群内部可以访问，这也是默认的 ServiceType。

  • NodePort： 通过每个 Node 上的 IP 和静态端口（NodePort）暴露服务。NodePort 服务会路由到 ClusterIP 服务，这个 ClusterIP 服务会自动创建。通过请求 :，可以从集群的外部访问一个 NodePort 服务。

  • LoadBalancer

    使用云提供商的负载局衡器，可以向外部暴露服务。外部的负载均衡器可以路由到 NodePort 服务和 ClusterIP 服务。仅作了解。

  • ExternalName

    通过返回 CNAME 和它的值，可以将服务映射到 externalName 字段的内容（例如，在集群内查找my-service.my-namespace.svc时，k8s DNS service只返回foo.bar.example.com这样的CNAME record）。没有任何类型代理被创建，网络流量发生在DNS层面。由于ExternalName要求kube-dns而我们使用的是coredns，也只作了解。

Ingress

Ingress是Kubernetes中的一个API对象，它管理外部访问集群内服务的路由规则。Ingress为服务提供了一个访问策略，允许外部客户端通过HTTP和HTTPS路由到集群内的不同服务。

实操

踩坑

  • YAML文件最好放在主工作目录下，避免篇频繁的路径切换

  • 注意很多地方命名不能使用下划线，尽量一开始就设计为中划线

  • 部署到云端服务器后，想通过浏览器访问，不要忘记配置云服务器的安全组

  • 出错时打印详细信息debug，不要只拿一个简陋的报错名就上网查

  • 要大幅度修改YAML文件或修改某些关键设置（比如修改pod名称），一定要先把原来的delete再修改YAML文件！不然原来的很多deployment取消不掉！

    • -L 选项让 curl 跟随重定向。

使用YAML文件管理Pod

通常情况下，我们不会直接使用kubectl run这样的命令行来管理容器，而是使用一个YAML格式的文件来描述Pod

  • 示例

  • 应用 YAML 文件对容器的管理

    将Pod的YAML描述保存到一个YAML文件中，如pod-demo.yaml，然后执行kubectl apply -f pod-demo.yaml，即可将该Podapply到K8s集群中

    apply的含义是，

    • 如果当前集群中不存在存在文件中对描述的资源，那将创建该资源；
    • 如果已经存在了，则根据YAML文件中的描述对资源进行更新。

    apply操作体现了K8s的声明式编程的思想

主要作用是努力使当前集群中的Pod数量与用户期望的状态相同。
